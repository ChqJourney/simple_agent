## Python 后端问题清单

### 高优先级

| 编号 | 问题 | 位置 | 描述 | 风险等级 |
|------|------|------|------|----------|
| PY-001 | **Shell 命令注入风险** | `tools/shell_execute.py` (line 209-290) | 虽然使用了 `shell_runner` 模式区分，但在非 Windows 系统下仍直接传递命令字符串给 shell | 高危 |
| PY-002 | **Python 代码执行风险** | `tools/python_execute.py` (line 46-112) | 直接执行用户提供的 Python 代码，没有沙箱隔离。可以访问文件系统、网络等系统资源 | 高危 |
| PY-003 | **文件路径遍历风险** | `tools/path_utils.py` (line 28-60) | `resolve_workspace_path` 函数虽然检查了路径是否在 workspace 内，但使用了 `resolve()` 可能绕过检查 | 中危 |
| PY-004 | **文件写入没有大小限制验证** | `tools/file_write.py` (line 89-97) | 虽然检查 `MAX_FILE_SIZE` (10MB)，但只是检查编码后的内容大小，可能导致磁盘空间耗尽 | 中危 |

### 中优先级

| 编号 | 问题 | 位置 | 描述 |
|------|------|------|------|
| PY-005 | **代码重复** | `llms/` 目录下 | 多个 provider 实现有大量重复代码，建议提取公共逻辑到基类 |
| PY-006 | **异常处理不一致** | 多处 | 有些位置使用 `try/except Exception` 过于宽泛，建议更精确地捕获特定异常类型 |
| PY-007 | **硬编码值较多** | 多个文件 | 存在魔法数字（如超时时间、缓冲区大小等），建议提取到配置文件中 |

### 低优先级

| 编号 | 问题 | 位置 | 描述 |
|------|------|------|------|
| PY-008 | **循环依赖风险** | `main.py` | 导入了大量模块，其中一些可能相互依赖，建议考虑使用依赖注入 |
| PY-009 | **WebSocket 连接状态管理复杂** | `main.py` | `runtime_state` 管理非常复杂，多个锁和异步操作增加死锁风险 |
| PY-010 | **配置验证分散** | `runtime/config.py` 和 `main.py` | 配置验证逻辑分散，建议集中配置验证逻辑 |

### 性能问题

| 编号 | 问题 | 位置 | 描述 |
|------|------|------|------|
| PY-011 | **文件 I/O 阻塞** | `Session` 类 | 多个方法使用同步文件 I/O，虽然使用了 `asyncio.to_thread` 包装，但仍可能成为瓶颈 |
| PY-012 | **内存使用** | `Agent._trim_messages_to_context_window` | 加载所有消息到内存，对于长会话可能占用大量内存 |

## 总结

**Python 后端：**
- 整体代码质量较高，架构设计合理
- 主要风险集中在工具执行的安全问题上
- 需要优先解决 Python 代码执行和 Shell 命令的安全沙箱化

**前端 & Tauri：**
- 架构清晰，使用现代技术栈（React + TypeScript + Zustand + Tauri）
- 主要问题是 WebSocket 认证竞态条件和文件访问安全
- 需要加强错误处理和边界情况处理

**作为 Windows 桌面单机程序，建议在发布前优先解决高优先级的安全问题（PY-001~003, FE-001~004）。**
